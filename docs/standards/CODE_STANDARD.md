# Code Standard

Главный документ репозитория. Имеет приоритет над всеми остальными стандартами.
Любой код в Genora Pro подчиняется этим правилам без исключений.

## Главное правило

Код всегда должен быть читаемым и понятным для junior-, middle- и senior-разработчиков
одновременно. Если фрагмент понятен только senior — он написан неверно.

Код не должен быть «умным». Код должен быть чистым, читаемым и корректным.

Требуется только чистый, читаемый и правильный код. Ничего сверх этого.

## Запрет комментариев

В коде не должно быть комментариев или подсказок, даже если они выглядят правильными
и уместными.

- Никаких пояснительных комментариев.
- Никаких TODO/FIXME/HACK.
- Никаких закомментированных блоков кода.
- Никаких баннеров, заголовков секций и ASCII-разметки.
- Никаких подсказок «что здесь происходит».

Если фрагмент требует комментария, чтобы быть понятным — это сигнал переписать фрагмент,
а не добавить комментарий. Понимание обеспечивается именами и структурой, а не текстом рядом.

Исключения только для машинно-обязательных директив, которые не являются пояснением:
`"use client"`, `"use server"`, прагмы линтера в редких обоснованных случаях.

## Что значит «не умный» код

- Прямолинейный поток вместо хитрых однострочников.
- Явные условия вместо вложенных тернарных операторов.
- Понятные промежуточные переменные вместо цепочек, которые нужно расшифровывать.
- Очевидное решение вместо «элегантного», которое экономит строки ценой ясности.

Оптимизация читаемости важнее краткости. Краткость не является целью.

## Именование

- Имена раскрывают намерение: что это и зачем, без сокращений-загадок.
- Функции — глаголы (`createOrder`, `getActiveSubscription`).
- Булевы значения — предикаты (`isActive`, `hasAccess`, `canCheckout`).
- Никаких `data`, `info`, `tmp`, `foo`, `x` как имён сущностей.
- Имя длиннее, но понятнее — всегда предпочтительнее.

## Структура

- Маленькие функции с одной ответственностью.
- Ранний возврат вместо глубокой вложенности.
- Минимум косвенности: не вводить абстракцию до того, как она реально нужна.
- Логика фичи живёт внутри фичи (см. `architecture/FEATURE_STRUCTURE.md`).

## Типизация

- TypeScript strict, без `any`.
- Без `@ts-ignore` и `@ts-expect-error` без явного обоснования в ADR.
- Типы описывают контракт; см. `standards/TYPESCRIPT_STANDARDS.md`.

## Критерий готовности кода

Код считается готовым, если:

- его читает junior и понимает без вопросов,
- в нём нет комментариев,
- он проходит `pnpm lint` и `pnpm typecheck`,
- решение прямолинейное, а не «умное».
